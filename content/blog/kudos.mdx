## Introduction: The Problem With Free Likes

Every major social media platform suffers from the same silent disease: **engagement inflation**. When liking something costs nothing — no money, no time, no cognitive load — the signal degrades to noise. A post with 10,000 likes might be genuinely brilliant, or it might just be relatable enough to trigger a reflexive double-tap.

**Kudos** was built on a radical premise: what if every act of appreciation had a real, measurable cost?

Not money — but *tokens*. A finite digital currency where every user starts with 100 and must spend thoughtfully, because once they're gone, they're gone (until the weekly stipend replenishes them). The result is a social platform where a post earning 50 tokens from 12 people carries far more signal than one with 10,000 "free" likes.

Building this required going far beyond typical CRUD architecture. Kudos is a polyglot microservices system with Go-powered performance cores, Node.js orchestration layers, atomic economic transactions, a custom feed-ranking algorithm inspired by OS scheduling theory, and automated background jobs that run a weekly economic cycle — taxation, redistribution, and content decay — every Sunday at midnight.

This post is the complete technical story.

## 1. System Architecture Overview

Kudos is composed of **8 independently deployable services** across two technology stacks, all orchestrated via Docker Compose and routed through a central API gateway.

### The Two-Stack Design Philosophy

The stack split wasn't arbitrary — it was driven by the nature of each service's workload.

**Node.js handles orchestration and I/O-bound work:** The API Gateway, Auth Service, Content Service, and User Service all run on Node.js. Node's non-blocking event loop is perfectly suited for the high concurrency of HTTP request routing, JWT validation, OTP delivery via Twilio, file uploads to S3/R2, and Prisma-based database queries that spend most of their time waiting on network I/O.

**Go handles computation and precision-critical operations:** The Token Service, Feed Service, and Jobs Service are written in Go. Token spending requires sub-millisecond atomic database transactions where a race condition means someone's balance goes negative. Feed ranking involves floating-point math on thousands of posts per request. The weekly economic cycle processes every account and post in the system in parallel. Go's goroutines, static typing, and predictable performance characteristics make it the right tool for this layer.

### Internal Communication: gRPC Over REST

Between the API Gateway and the two Go services, all communication happens over **gRPC** — not HTTP/REST. This decision was deliberate.

gRPC uses HTTP/2 with binary Protocol Buffer serialization, giving roughly 5–10x the throughput of JSON over HTTP/1.1 for high-frequency calls. Since every page load hits the Token Service (to show balance) and Feed Service (to render posts), shaving latency here has a multiplying effect on user experience.

The `.proto` contracts also serve as strict API contracts between Node.js and Go — no more "is that field `camelCase` or `snake_case`?" debates.

```protobuf
// proto/token.proto
service TokenService {
  rpc SpendTokens (TokenSpendRequest) returns (TokenSpendResponse);
  rpc GetBalance (TokenBalanceRequest) returns (TokenBalanceResponse);
  rpc GetCommentCost (CommentCostRequest) returns (CommentCostResponse);
  // ...
}

message TokenSpendRequest {
  string account_id = 1;
  optional string page_id = 2;
  string target_type = 3;  // "post" or "comment"
  string target_id = 4;
  double amount = 5;
}
```

## 2. The Token Economy: Math-Driven Fairness

The economic model is the philosophical heart of Kudos. Every design decision traces back to one goal: make engagement meaningful by making it costly.

### Initial Allocation and Inflows

Every new account receives **100 tokens** on creation — enough to explore, discover creators, and support a few posts before needing to earn more. The only ongoing inflows are:

1. **Creator earnings** — 60% of tokens spent on your content
2. **Weekly stipends** — distributed from the government pool (more on this below)

### The 60/20/20 Atomic Split

When a user spends tokens on a post, the distribution happens in a single PostgreSQL transaction that touches four rows across multiple entities:

```go
// internal/token/service.go
func (s *Service) SpendTokens(ctx context.Context, req *SpendTokensRequest) (*SpendTokensResponse, error) {
    // Validate, check duplicate, prevent self-spending...

    tx, err := s.db.Begin(ctx)
    defer tx.Rollback(ctx) // Safety net

    // Lock both accounts to prevent race conditions
    spenderBalance, _ := s.repo.GetAccountBalance(ctx, tx, req.AccountID)     // FOR UPDATE
    creatorBalance, _ := s.repo.GetAccountBalance(ctx, tx, creatorAccountID)  // FOR UPDATE

    creatorShare := req.Amount * 0.60  // 60% to creator
    govShare    := req.Amount * 0.20  // 20% to government pool
    burnShare   := req.Amount * 0.20  // 20% permanently burned

    s.repo.UpdateAccountBalance(ctx, tx, req.AccountID, spenderBalance - req.Amount)
    s.repo.UpdateAccountBalance(ctx, tx, creatorAccountID, creatorBalance + creatorShare)
    // govShare is recorded in system_state.government_pool
    // burnShare increments system_state.burned_total — gone forever

    s.repo.CreateTransaction(ctx, tx, &Transaction{...})
    tx.Commit(ctx)
}
```

The burn mechanism (20% of every transaction) is deflationary — it permanently removes tokens from circulation, preventing hyperinflation as new users join. The government pool (another 20%) funds the weekly redistribution cycle.

### Row-Level Locking and Race Conditions

A subtle but critical detail: we use `SELECT ... FOR UPDATE` to lock both the spender and creator account rows before modifying them. Without this, two simultaneous transactions could both read the same balance, both decide there's enough to spend, and both commit — resulting in a negative balance.

For accounts with multiple "pages" (personas), we use the **account ID** as the lock key, not the page ID. This prevents a scenario where two different pages on the same account spend tokens simultaneously and overdraw the shared pool.

### Progressive Taxation

Wealth concentration is the silent killer of token economies. If early adopters accumulate thousands of tokens while new users struggle with their initial 100, the platform stratifies into haves and have-nots. Kudos addresses this with a weekly progressive tax:

| Balance Range | Tax Rate | Applied To |
| :------------ | :------- | :--------- |
| 0 – 200       | 0%       | Exempt     |
| 201 – 500     | 5%       | Amount > 200 |
| 501 – 1,000   | 10%      | Amount > 500 |
| 1,001 – 2,000 | 15%      | Amount > 1,000 |
| 2,001+        | 20%      | Amount > 2,000 |

A user holding 1,500 tokens pays:
```
  0% × 200 =  0.00
  5% × 300 = 15.00
 10% × 500 = 50.00
 15% × 500 = 75.00
─────────────────
Total tax  = 140 tokens/week
```

This tax runs every Sunday at midnight UTC, processes accounts in parallel via goroutines with a semaphore-bounded concurrency limit of 50, and feeds every collected token directly into the government pool.

### Universal Basic Income (Stipends)

Taxes collected aren't destroyed — they're redistributed. Every Sunday at 1 AM (one hour after tax collection), the Jobs Service runs a stipend distribution:

```
Base Stipend = (Government Pool × 0.80) / Active Accounts
+ Engagement Bonus: +10 tokens if you earned any tokens last week
+ New User Bonus:   +20 tokens for first 4 weeks after signup
```

This creates a feedback loop: active creators earn more through both direct payments AND higher stipends. New users get extra runway to establish themselves. The system self-corrects toward activity rather than passive holding.

### The Unlike Window

One of the more unusual UX decisions: users have a **5-minute window to unlike** a post and receive a full refund. This was inspired by the "undo send" pattern — it acknowledges that people sometimes fat-finger a spend, but prevents gaming (you can't unlike after watching engagement metrics).

When unlike happens, the entire transaction is reversed atomically: tokens returned to spender, deducted from creator, content metrics updated, transaction marked `unliked_at`.

## 3. The Feed Algorithm: Multi-Level Feedback Queues

Most social feeds use some variant of `(engagement * weight) / time_decay`. Kudos takes inspiration from an unlikely source: **operating system process scheduling**.

### MLFQ: A Brief Primer

Multi-Level Feedback Queue schedulers (used in macOS, Linux, and most modern OSes) maintain multiple priority queues. New processes start in the highest-priority queue. If they use their entire time slice (i.e., they're CPU-intensive), they get demoted to a lower queue. Short, interactive jobs naturally float to the top.

We apply the same logic to content. New posts start in Queue 0 (Hot). If they keep receiving tokens, they stay hot. If engagement drops, they demote. The system learns which content is genuinely valuable versus momentarily viral.

### Queue Definitions

| Queue | Name | Visibility | Promotion Condition | Demotion Condition |
| :---- | :--- | :--------- | :------------------ | :----------------- |
| 0 | Hot | 2.0× | Already highest | velocity < 0.1 OR age > 24h |
| 1 | Warm | 1.5× | velocity > 2.0 token/hr | velocity < 0.1 OR age > 72h |
| 2 | Cold | 1.0× | velocity > 1.0 token/hr | age > 168h OR no tokens in 7 days |
| 3 | Frozen | 0.3× | N/A — archive | N/A — terminal |

The queue management job runs **every hour** via the Go scheduler. It pulls all non-frozen posts, calculates their current engagement velocity, and applies the transition rules:

```go
// internal/jobs/queue.go
func (s *QueueService) processHotQueue(ctx context.Context, post PostInQueue, 
    hoursInQueue, velocity float64) (string, error) {
    
    if hoursInQueue > HotQueueMaxHours || velocity < DemotionThreshold {
        return s.demotePost(ctx, post.PostID, QueueWarm)
    }
    return s.updateVisibilityScore(ctx, post.PostID, velocity)
}
```

### The Visibility Score Formula

Within each queue, posts are ranked by a composite visibility score:

```
visibility_score = (token_score × 0.5) + (recency × 0.3) + (velocity × 0.2)
```

Where:
- `token_score = log(1 + total_tokens)` — logarithmic to prevent runaway accumulation
- `recency = 1 / (1 + hours_old / 24)` — smooth decay, halves every 24 hours
- `velocity = tokens / hours_since_creation` — rewards recent engagement bursts

The queue-level multiplier then amplifies this score: a post in Queue 0 with score 10 appears above a Queue 1 post with score 20 in the default Hot feed.

### Token Decay: Keeping Content Fresh

Posts don't just age out of queues — their token balances also decay over time. If a post receives no new tokens for 7+ days:

```
decay_amount = current_tokens × decay_rate
```

The default decay rate is 2% per week for inactive content. Active content (receiving tokens regularly) has a near-zero decay rate — it's self-sustaining.

Decayed tokens flow back to the government pool, not into the void. This recirculates "locked up" economic value from old content back into the active economy.

```go
// internal/jobs/decay.go
func (s *DecayService) applyDecayToPost(ctx context.Context, post PostForDecay) (float64, bool, error) {
    daysSinceLastToken := int(time.Since(*post.LastTokenReceivedAt).Hours() / 24)
    if daysSinceLastToken < 7 { return 0, false, nil } // Not yet eligible
    
    decayAmount := post.TotalTokens * post.DecayRate
    newBalance := post.TotalTokens - decayAmount
    shouldArchive := newBalance < 1.0 // Archive if less than 1 token remains
    
    // Atomic update: post balance + system state in one transaction
    tx.Exec(ctx, updatePostQuery, newBalance, decayAmount, shouldArchive, post.PostID)
    tx.Exec(ctx, createDecayEventQuery, ...)
    tx.Commit(ctx)
    
    return decayAmount, shouldArchive, nil
}
```

## 4. The Database Schema: Designing for Economic Integrity

The PostgreSQL schema was designed with economic correctness as the primary constraint — not convenience.

### Core Entities

**`accounts`** — One per phone number. Holds the canonical token balance, lifetime earnings/spending, tax history, and stipend records. The phone number constraint (`CONSTRAINT chk_phone_format CHECK (phone_number ~ '^\+[1-9]\d{1,14}$')`) prevents mass account creation.

**`pages`** — User-facing identities (usernames, avatars, bios). Accounts can have multiple pages in theory (different personas), but all pages share the same account token pool. This prevents gaming via fake account multiplication.

**`token_transactions`** — Every token movement is recorded. This table is append-only and includes a constraint ensuring the amounts always add up:
```sql
CONSTRAINT chk_amounts_sum CHECK (
    tokens_spent = tokens_to_creator + tokens_to_government + tokens_burned
)
```

**`system_state`** — A singleton table (enforced by `CHECK (id = 1)`) tracking global economic metrics: total circulation, government pool balance, total burned, last job run timestamps.

### Indexes Designed for Query Patterns

The feed query is the highest-frequency read in the system. Its index was designed around the exact query pattern:

```sql
-- Hot Feed: Queue 0-1, sorted by visibility score
CREATE INDEX idx_posts_hot_feed ON posts(queue_level, visibility_score DESC, created_at DESC)
    WHERE deleted_at IS NULL AND queue_level IN (0, 1);

-- Unlike window queries (checked on every spend)
CREATE INDEX idx_transactions_unlikeable_window
    ON token_transactions(spender_account_id, can_unlike_until, transaction_id)
    WHERE unliked_at IS NULL;

-- Tax collection: only accounts above the exempt threshold
CREATE INDEX idx_accounts_tax_eligible
    ON accounts(last_tax_collected_at, token_balance)
    WHERE token_balance >= 200;
```

The partial indexes (`WHERE deleted_at IS NULL`) dramatically reduce index size for soft-deleted records and keep hot-path queries fast as data accumulates.

### Views for Reporting

Two materialized view patterns handle common reporting queries without re-joining large tables:

```sql
-- v_hot_feed: pre-joins posts with page info for the feed
CREATE VIEW v_hot_feed AS
SELECT p.post_id, p.content, p.total_tokens_received,
       p.unique_spenders_count, p.visibility_score, ...
       pg.username, pg.display_name, pg.avatar_url
FROM posts p JOIN pages pg ON p.page_id = pg.page_id
WHERE p.deleted_at IS NULL AND p.queue_level IN (0, 1)
ORDER BY p.visibility_score DESC;

-- v_account_wealth: includes wealth rank for leaderboards
CREATE VIEW v_account_wealth AS
SELECT a.account_id, a.token_balance, ...,
       RANK() OVER (ORDER BY a.token_balance DESC) as wealth_rank
FROM accounts a LEFT JOIN posts p ON a.account_id = p.account_id;
```

## 5. The API Gateway: Single Entry Point

All 8 backend services expose a single public port: **8080** on the API Gateway. Clients never talk directly to Auth, Content, or Go services.

The gateway handles:

1. **JWT Authentication** — Validates tokens on every protected route, attaches `accountId` and `phone` to the request context
2. **Rate Limiting** — 100 requests per 15-minute window (general), 20/minute for token spending, 5/15-minutes for auth attempts
3. **Request ID Injection** — Every request gets a `X-Request-ID` header for distributed tracing
4. **gRPC Bridging** — Token and Feed routes are translated from HTTP/JSON to gRPC/Protobuf internally
5. **HTTP Proxying** — Auth, Content, and User routes are proxied via Axios with configurable timeouts
6. **Error Normalization** — gRPC status codes (FAILED_PRECONDITION, NOT_FOUND, etc.) are mapped to appropriate HTTP status codes with user-friendly messages

One important architectural decision: the gateway **pre-charges tokens before creating content**. When a user posts or comments, the gateway first calls the Token Service to debit the account. Only if that succeeds does it proxy the request to the Content Service. This prevents content creation without payment, even if the Content Service is unavailable during the debit phase.

```typescript
// api-gateway/src/routes/proxy.routes.ts
router.post("/posts", authenticateToken, async (req, res, next) => {
    // Step 1: Charge for post (deducts 1 token or uses free daily post)
    const chargeResult = await tokenClient.chargePost({ account_id: accountId });
    
    // Step 2: Only create post if charge succeeded
    const response = await httpClient.post(
        `${CONTENT_SERVICE_URL}/api/v1/posts`, req.body, 
        { headers: { Authorization: req.headers.authorization } }
    );
    res.status(response.status).json(response.data);
});
```

## 6. The Frontend: Next.js with Zustand and React Query

The frontend is a Next.js 16 application with a neobrutalist design aesthetic — bold borders, offset shadows, and high contrast — reflecting the platform's philosophy of giving engagement more weight.

### State Management Architecture

Three Zustand stores handle client-side state:

**`auth.store.ts`** — Persisted to localStorage via `zustand/middleware`. Holds JWT token, user profile, and page data. Uses Zustand's `persist` middleware with a custom `partialize` function to prevent ephemeral state from being written to disk.

**`tokens.store.ts`** — In-memory token balance with optimistic update helpers. When a user spends tokens, the balance updates immediately (optimistic) before the API call returns — preventing the jarring "balance changed" flash.

**`ui.store.ts`** — Toast notifications, modal state, and sidebar toggle. Toasts auto-dismiss via `setTimeout` in the store action itself.

### Data Fetching with React Query

All server state goes through TanStack Query with a structured key hierarchy:

```typescript
export const queryKeys = {
  posts: {
    detail: (id: string) => ["posts", "detail", id] as const,
    userPosts: (username: string) => ["posts", "user", username] as const,
  },
  feed: {
    global: (filters?: object) => ["feed", "global", filters] as const,
    trending: (timeWindow: string) => ["feed", "trending", { timeWindow }] as const,
  },
  tokens: {
    balance: ["tokens", "balance"] as const,
    commentCost: ["tokens", "comment-cost"] as const,
  }
}
```

The key hierarchy ensures granular cache invalidation: spending tokens on a post invalidates `posts.detail(postId)`, `tokens.balance`, and `tokens.transactions` — but not the entire feed.

### The Spend Dialog UX

The token spend dialog shows a real-time breakdown of where tokens go, updating as the slider moves:

## 7. The Jobs Service: Automated Economic Governance

The most unusual component of Kudos is the Jobs Service — a Go process that runs no user-facing HTTP routes, only scheduled background work that keeps the economic cycle turning.

### Scheduler Architecture

The scheduler uses Go's `time.NewTicker` and `time.AfterFunc` to run four independent goroutines:

| Job | Schedule | Purpose |
| :-- | :------- | :------ |
| Decay | Daily at 2 AM UTC | Apply 2% weekly decay to inactive posts |
| Tax Collection | Every Sunday 12 AM UTC | Progressive taxation on wealthy accounts |
| Stipend Distribution | Every Sunday 1 AM UTC | Redistribute taxes to active users |
| Queue Management | Every hour | Promote/demote posts between MLFQ tiers |

Each job is idempotent and checks its last-run timestamp from `system_state` before executing, preventing duplicate processing if the service restarts.

### Concurrency Pattern

All four jobs use the same concurrency pattern: a worker pool bounded by a semaphore, with results accumulated into a stats struct protected by a mutex:

```go
const maxConcurrent = 50
sem := make(chan struct{}, maxConcurrent)
var wg sync.WaitGroup
var mu sync.Mutex

for _, account := range accounts {
    wg.Add(1)
    sem <- struct{}{} // Acquire semaphore (blocks if pool is full)
    
    go func(acc TaxableAccount) {
        defer wg.Done()
        defer func() { <-sem }() // Release semaphore
        
        tokensCollected, err := s.collectTaxFromAccount(ctx, acc)
        
        mu.Lock()
        stats.TotalCollected += tokensCollected
        mu.Unlock()
    }(account)
}
wg.Wait()
```

This pattern processes up to 50 accounts simultaneously without overwhelming the database connection pool (capped at 50 connections for the Jobs Service, configured via `pgxpool`).

### Database Connection Pool Configuration

The Go services use `pgx/v5` with a custom pool configuration. Notably, we use `QueryExecModeDescribeExec` to avoid prepared statement conflicts across connection pool members:

```go
// internal/db/postgres.go
config.ConnConfig.DefaultQueryExecMode = pgx.QueryExecModeDescribeExec

config.BeforeAcquire = func(ctx context.Context, conn *pgx.Conn) bool {
    if err := conn.Ping(ctx); err != nil {
        return false // Discard unhealthy connections
    }
    return true
}
```

The Jobs Service gets a larger pool (`MaxConns: 50`) than API services (`MaxConns: 25`) because it processes batches of hundreds of rows simultaneously.

## 8. Authentication: Phone-First Identity

Kudos uses phone number verification (via Twilio) instead of email/password for two deliberate reasons:

1. **One account per person** — Phone numbers are significantly harder to mass-create than email addresses, which limits Sybil attacks
2. **No password management** — No forgotten passwords, no password resets, no credential stuffing

The auth flow uses a stateless OTP approach:

```
1. POST /auth/send-otp { phone: "+1234567890" }
   → Creates phone_verifications record with SHA-256 hashed OTP
   → Sends OTP via Twilio
   → Returns verification_id

2. POST /auth/verify-otp { verification_id, code }
   → Validates code, checks expiry (5 minutes), checks attempts (max 3)
   → Creates or retrieves Account record
   → Returns JWT (1hr access) + refresh token (7d)
   → Access token payload: { accountId, phone }
```

All downstream services validate JWTs using the same `JWT_SECRET` environment variable — no inter-service auth calls required. The Auth Service is only contacted for the initial OTP flow.

## 9. Content Economics: Comments and Posts

Content creation has an economic model too, not just token spending.

**Posts:** The first post of the day is free. Subsequent posts cost 1 token. This prevents spam while not blocking users who've run dry.

**Comments:** The first 2 comments per day are free. Additional comments cost 0.5 tokens. This encourages discussion without turning it into a pay-wall.

The free-post and free-comment counters reset daily (tracked in `accounts.daily_comments_reset_at`) and are checked atomically in the Token Service before content is created.

Comment token spending has a different distribution than post spending:
- 30% → original post creator (rewards content that sparks discussion)
- 10% → commenter (if their comment receives tokens from others)
- 10% → government pool
- 50% burned (stronger deflationary pressure, since comments are lower-value than posts)

## 10. Observability and Performance

### Performance Benchmarks

In testing, we hit these targets:

| Operation | Latency |
| :-------- | :------ |
| Token Spend (incl. DB transaction) | < 40ms |
| Global Feed (20 posts) | ~50ms |
| Personalized Feed (collaborative filtering) | ~150ms |
| Get Balance | ~10ms |

The personalized feed's 150ms comes from two sequential queries: first finding users with similar spending patterns, then fetching posts those users engaged with that the current user hasn't seen yet. This is a simplified collaborative filtering approach — cosine similarity over spending history rather than a full ML model.

## 11. What We Learned

### Social media is a distributed systems and economics problem

The hardest bugs weren't in the code — they were in the model. What happens to decayed tokens? (Government pool, not burned — we want recirculation.) How do we prevent two simultaneous spends overdrawing an account? (Row-level locks with `FOR UPDATE`.) What if a user likes their own content via an alt page? (The account-level unified pool means it's economically neutral — you'd just be transferring tokens to yourself.)

### gRPC's ergonomics are rough but worth it

The protobuf setup, code generation, and error mapping between gRPC status codes and HTTP status codes added significant boilerplate. But the structured contracts and binary efficiency paid dividends: the gateway's gRPC client code is strictly typed, and adding a new RPC endpoint requires updating both the `.proto` file and the handler — you physically cannot forget one.

### Zustand + React Query is the right frontend stack for this

We evaluated Redux, Jotai, and SWR. Zustand handles client-only state (auth, tokens, UI) elegantly without boilerplate. React Query handles server state (feed, posts, comments) with automatic caching, background refetching, and optimistic updates. The key insight: don't put server state in client state managers. They have different lifecycles.

### Economic design is harder than technical design

The tax brackets, decay rates, stipend formulas, and burn percentages all interact in non-obvious ways. We ran simulations before implementation: what happens in 6 months if 10,000 users join? Does the government pool accumulate to the point where stipends become meaninglessly small? (Yes, if decay rates are too low — decay feeds the pool, so it needs to be balanced against pool distribution speed.) These are game theory problems dressed up in TypeScript.

**Federated Hubs** — Allow different organizations (universities, companies, communities) to run Kudos clusters with their own token economies while sharing identity and optionally allowing cross-hub spending.

**Token Marketplace** — Official mechanism for gifting tokens between accounts, with transparent transaction history. Black markets will exist regardless; making the official version good enough to use is better than ignoring it.

**ML-Powered Feed** — Replace the simplified collaborative filtering with a proper embedding-based recommendation model, trained on spending patterns rather than clicks.

**Verifiable Token Supply** — Publish a real-time cryptographic audit of the token economy: total created, burned, in circulation, in government pool. Maximum transparency.

---

Building Kudos taught us that you can't bolt economics onto a social platform as an afterthought. The economic model has to be the architecture. Every table, every index, every service boundary was shaped by the question: *what ensures the economy stays fair?*

When engagement costs something real, it means something real. That's the whole bet.
